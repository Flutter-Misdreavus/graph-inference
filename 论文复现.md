# Graph-as-Code 论文复现计划 —— Arxiv 数据集

## 1. 项目概述

复现论文《Actions Speak Louder Than Prompts》中 **Graph-as-Code (φ_code)** 交互模式在 **Arxiv 数据集** 上的节点分类 pipeline。

- **目标数据集**: Arxiv (论文中 short-text homophilic, 40类, 169,343节点, 1,166,243边)
- **目标模型**: DeepSeek / Kimi (API 调用)
- **论文基准**: 74.40±3.02% (Table 1)
- **复现深度**: 可运行的 end-to-end pipeline，输入节点 → LLM 推理 → 输出预测标签

---

## 2. 数据准备阶段

### 2.1 现有数据盘点
- `Datas/Arxiv.csv`: 节点文本特征(title, abstract, text) + label_id (40类)
- `Datas/Feature/Arxiv_roberta_base_512_cls.npy`: 编码特征（本复现不使用，Graph-as-Code 直接用原始文本）
- **缺失**: 图结构（边/邻接关系）

### 2.2 图结构数据获取
- 从 **OGB (Open Graph Benchmark)** 下载 `ogbn-arxiv` 的边列表
- 边为有向边（论文引用关系），构建出无向邻居列表
- 数据分割沿用论文标注: Train/Val/Test = 53.7%/17.6%/28.7%

### 2.3 数据预处理
- 构建 `df` DataFrame，每行对应一个节点：
  - `node_id`: 节点索引
  - `features`: 文本特征（title + abstract）
  - `neighbors`: 邻居节点 ID 列表
  - `label`: 训练集标签 (int)，测试集为 None
- 保存为 pickle/csv 供 pipeline 加载

## 3. Graph-as-Code Pipeline 实现

### 3.1 核心交互循环 (ReAct 风格)

```
初始化: 提供 task prompt + DataFrame schema + 类别定义

循环直到模型输出 Answer:
  1. LLM 接收当前上下文（历史交互记录）
  2. LLM 进行 reasoning，生成一条 pandas/python 查询代码
  3. 环境沙箱执行代码，返回执行结果
  4. 将结果追加到交互历史
  5. 若 LLM 输出 "Answer: [class_id]"，则终止循环
```

### 3.2 Prompt 模板 (参照论文 Template 3)

- **Task**: 为节点 node_id 分类
- **Schema**: DataFrame 结构说明（features, neighbors, label 三列）
- **可用类别**: 40个 arxiv 类别编号 + 名称
- **Response 格式**: 
  - 中间步骤: reasoning + 单行 pandas 表达式
  - 终止: `Answer: [class_id]`
- **约束**: 避免打印整个 DataFrame（数据量大）

### 3.3 代码执行环境

- 使用 **受限 Python 沙箱** 执行 LLM 生成的代码
- 仅暴露: `df` (DataFrame), `pd` (pandas), 基础内置函数
- 禁止: 网络访问、文件系统写操作、导入非白名单模块
- 超时控制: 单步执行不超过 5 秒

### 3.4 典型推理路径（论文 Example 1 风格）

```
Step 1: 查看目标节点的 features 和 label（训练集为 None）
Step 2: 查看邻居节点的 labels，做多数投票
Step 3: 查看邻居的 features 验证假设
Step 4: 综合判断，输出 Answer
```

## 4. API 集成层

### 4.1 多模型支持
- 统一接口封装 DeepSeek 和 Kimi API
- 配置文件管理 API Key 和模型参数（temperature, max_tokens 等）

### 4.2 调用参数
- **temperature**: 低值（如 0.1~0.3），保证推理确定性
- **max_tokens**: 足够大以容纳长 reasoning + 代码
- **system prompt**: 固定为 Graph-as-Code 的 task 描述

### 4.3 容错与重试
- API 调用失败自动重试（指数退避）
- LLM 输出格式异常时，prompt 修正后重试
- 执行代码报错时，将错误信息反馈给 LLM 让其修正

## 5. 评估与输出

### 5.1 评估指标
- **准确率 (Accuracy)**: 预测正确数 / 总测试数
- 论文基准: 74.40±3.02%（多次 seed 平均）

### 5.2 评估流程
- 从 Test 集随机采样 N 个节点（默认 100，可配置）
- 对每个节点运行完整 Graph-as-Code pipeline
- 收集预测结果与真实标签对比
- 输出准确率 + 混淆矩阵（可选）

### 5.3 日志与调试
- 每个节点的完整交互历史（reasoning + code + result）保存到日志
- 支持可视化单节点的推理轨迹（用于调试和分析）

## 9. 风险与注意事项

1. **Token 成本**: Arxiv 平均度 13.64，Graph-as-Code 会多次调用 API。建议先在小样本（如 10-20 个节点）上验证 pipeline，再扩大规模。
2. **API Key 安全**: 使用 `.env` 文件存储 Key，不提交到代码仓库。
3. **代码执行安全**: 沙箱必须严格限制，防止 LLM 生成恶意代码。
4. **论文分割差异**: OGB 的默认分割可能与论文不完全一致，如结果偏差较大需排查此因素。
5. **Kimi API 兼容性**: Kimi 的 API 格式可能与 OpenAI 兼容，需实际验证工具调用和代码生成效果。
